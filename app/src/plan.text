Plan: Reactive UI Update System
A reactive UI architecture using retained mode with dirty tracking and event-driven data bindings, 
optimizing both mesh and text rendering through incremental updates.

Steps
Implement dirty tracking system — Add DirtyFlags { layout, mesh, text } to WidgetBase, 
create mark_dirty() and propagate_dirty_to_parent() methods, extend Widget trait with is_dirty() 
and clear_dirty() queries. Track three dirty states independently since layout changes don't always 
affect rendering.

Create event propagation infrastructure — Build UiEventRouter in new base/src/ui/events.rs 
with hit-testing using widget rects from layout, implement event bubbling (child → parent) 
with EventResponse::Handled/Continue, add Widget::on_event(&mut self, event: UiEvent) -> EventResponse 
trait method for keyboard/mouse/focus events, maintain focus stack for keyboard routing.

Add observable data binding — Create Observable<T> wrapper in base/src/property/observable.rs 
using Arc<Mutex<ObservableInner>> with subscriber callbacks, implement PropertyPanel::subscribe_to_changes() 
that registers dirty-marking callbacks on field changes via Propertied::set_value(), store widget-to-data 
subscriptions in new Scene::bindings field.

Implement incremental collection with caching — Add CollectionCache { meshes: Vec<Mesh>, 
text: TextParams, version: u64 } to Scene, modify collect_rects()/collect_text() to skip 
subtrees where !widget.is_dirty_mesh() / !widget.is_dirty_text(), cache Buffer instances in
TextRenderer with HashMap<WidgetId, CachedBuffer> comparing widget versions (extends Step 1 
from previous text caching plan), only call UiMeshBuilder::build() on dirty widget branches.

Integrate with frame loop — Modify Scene::update() to process input events via 
UiEventRouter::route_events(input, &mut widget_tree), check if any widgets are dirty before running 
incremental collection in update_passes(), propagate dirty flags from data changes 
(triggered by event handlers like TextField::on_event modifying bound data via Propertied::set_value), 
clear dirty flags after successful render.

Optimize PropertyPanel rebuilding — Replace full reconstruction with PropertyPanel::refresh() 
that diffs Propertied::get_value() results against cached values, implement smart partial rebuilds: 
for PropertyNode::Leaf changes only update the corresponding TextField widget, for PropertyNode::Group 
structure changes rebuild only that subtree, maintain stable WidgetIds using semantic keys (property hash) 
instead of instance-based generation.

Further Considerations
Widget ID stability strategy — Current IDs are instance-based (lost on rebuild). Options: A) 
Semantic IDs from stable keys (property name hash for PropertyPanel fields, 
user-provided keys for dynamic lists), B) Reconciliation algorithm matching widgets by 
type+position (React-like), C) Explicit key prop system. Recommendation?

Layout cache invalidation rules — When does cached layout become invalid? Propose: 
invalidate on parent size change, self size change, style param change (LayoutParams version counter), 
child addition/removal. Should layout results be cached in WidgetBase or separate LayoutCache structure?

Observable granularity trade-off — Fine-grained observables (per-field subscriptions via 
Propertied::subscribe(key: u64, callback)) vs coarse-grained (whole object dirty on any set_value()). 
Fine-grained minimizes re-renders but adds subscription overhead. Which approach for initial implementation?

GPU buffer update strategy — For meshes: single dynamic buffer with sub-buffer updates vs 
per-widget static buffers? For text: current HashMap<WidgetId, Buffer> approach confirmed? 
Need double-buffering to avoid GPU stalls during partial updates?


Plan: Property-Driven Change Collection
A centralized change collection system using property keys as stable identifiers, 
capturing mutations at the PropertySchema::try_set_from_str boundary to avoid UI tree 
scanning and trait pollution.

Steps
 

Thread ChangeSet through PropertyPanel — Add ChangeSet field to PropertyPanel<T, C>, 
create PropertyPanel::apply_text_edit(widget_id: WidgetId, text: String) -> 
Result<(), PropertyError> method that finds corresponding TextField by widget_id, 
looks up its property key from build-time mapping, calls 
schema.try_set_from_str(&mut locked_obj, &text, &system, &mut self.changes), 
store HashMap<WidgetId, u64> mapping TextField widget IDs to property keys during 
build_into_grid().

Map property keys to affected UI regions — Create DependencyGraph<C: PropertyConfig> 
in base/src/property/deps.rs with register_dependency(property_key: u64, widget_id: WidgetId) 
and get_affected_widgets(key: u64) -> Vec<WidgetId>, populate during PropertyPanel 
build (one TextField per property = 1:1 mapping initially), extend for computed 
properties later (e.g., line length depends on both endpoint coordinates, so I_KEY 
and J_KEY affect same Label widget).

Integrate with event system from Plan step 2 — In TextField::on_event(KeyPress) 
handler, accumulate typed characters into local buffer, on Enter/FocusLost emit UiEvent::TextCommitted(widget_id, text), Scene routes to PropertyPanel via panel.apply_text_edit(widget_id, text), after all events processed call panel.drain_changes() to get ChangeSet, use DependencyGraph::get_affected_widgets() to mark only affected widgets dirty for re-collection.

Optimize incremental updates — When ChangeSet reports property key changed (e.g., 
PointData::X_KEY), only invalidate: the specific TextField showing that value (dirty text), 
the PropertyPanel's mesh (dirty mesh for TextField background), any computed displays 
using that value, skip re-layout if only value changed (not structure), skip unrelated 
widgets entirely even within same PropertyPanel.

Further Considerations
ChangeSet ownership and lifetime — Should ChangeSet be: A) Owned by PropertyPanel
 (each panel tracks own changes), B) Global per Scene (centralized), C) Thread-local 
 (multi-scene support)? Trade-off: isolation vs global coordination.

Batching vs immediate propagation — Current plan batches changes (collect during event 
processing, apply after). Alternative: immediately mark dirty when record_change() 
called. Batching allows de-duplication (multiple edits to same property) but delays 
invalidation. Preference?

Change granularity for computed properties — Example: Line length = distance(point_i, 
point_j). When PointData::X_KEY changes, must invalidate line's length display. 
Options: A) Explicit DependencyGraph with manual registration, 
B) Automatic tracking via getter instrumentation (like reactive frameworks), 
C) Whole-object invalidation (any Point property → all Lines). Which complexity level?

Validation and rollback — If try_set_from_str returns Err (invalid input), should we: 
A) Not record change in ChangeSet, B) Record failed attempt for error display, 
C) Support undo via ChangeSet::revert()? Need error UI feedback mechanism.
